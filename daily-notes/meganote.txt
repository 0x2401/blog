HTTP is a stateless protocol, meaning every request is independent of the previous one. From the server's perspective, three HTTP requests A, B, and C from the same user may as well be from three different users. This is a problem for applications that need to maintain consistent state across multiple requests, such as every single website with accounts ever. There are two ways of solving this problem:

1. Sessions: You log in with your username and password. A server creates an entry in a database with a random string of characters as a unique ID and gives it to you. You provide that ID on every subsequent request, which the server uses to query the session table and recognize you. This is easy and simple to implement. The downside is that the server has to do a database query for every single request it receives, either from you or from other users. Also, if anyone ever steals that session ID, you're cooked.

2. Tokens: Same approach as above, but trash the database. Replace it with a special token or document, signed by the server so you know it's legit. Every time you issue an authenticated request, provide the token. The mere fact of you having the token (ideally) means I know the claims contained within it (that you are who you say you are) are trustworthy.

Docker is a containerization platform. Docker solves the problem of an application working on one computer, but not any others due to differences in installed dependencies, programming language/runtime versions, libraries, etc. Containers are isolated computing environments with their own networking, process table, and filesystem. Unlike VMs, containers do not have their own OS/kernel. Instead, multiple containers on a single host share an operating system kernel, managed by a container runtime, but isolated from each other.

Write-Ahead Logs
Databases must keep data in main memory (RAM) for quick access. However, RAM is volatile - in the event of a crash or power interruption all data and any associated changes are lost. Secondary disk memory is stable, but at the cost of significantly slower access, making it unsuitable for performance reasons. Databases solve this my storing active data in main memory in the form of pages that they occasionally sync down to disk storage. However, this process can be slow due to varying data change sizes between commits and their distribution across the main memory address space. Additionally, power interruption or a crash while a page is being synced to stable storage can cause data corruption.

Write-ahead logging (WAL) resolves both of these problems. A write-ahead log is an append-only list that lives on disk for crash recovery. Every operation that modifies data must be logged to the disk before any changes can be made to the associated pages. This means that database transactions are effectively committed by committing log records, not the entire pages themselves, saving what would be expensive fsync() operations for a later date while preserving durability. If the system crashes after the log entry but before the actual pages are saved, the system can reconstruct their state in memory on restart by simply reading the WAL. In order to do this, two rules must be enforced:
- The undo rule means that no dirty page can be sent to disk before its associated log record.
- The redo rule means all log records for a transaction must be saved to disk before confirmation of the commit is sent to the client.
These two characteristics mean that WAL directly provides durability and atomicity, two ACID properties. The redo rule provides durability, because it ensures that no commit will ever be acknowledged without the data on disk needed to reconstruct it in the event of system failure. The undo rule provides atomicity, because the log records contain data about the previous state of the data, as well as the new state, allowing partially completed transactions to be rolled back using the log information.

1. Early mechanical concept of a programmable machine (1830s–1840s)

Charles Babbage: Designed the Analytical Engine, with a “store” (memory), “mill” (processor), and input/output. This is the first complete conceptual model of a general-purpose computer.
Ada Lovelace: Described how such a machine could execute sequences of instructions and manipulate symbols. She articulated the idea of a program.
Source: Encyclopaedia Britannica, “Analytical Engine.”

2. Mathematical definition of computation (1930s–1940s)

Alan Turing: Defined computation formally with the Turing machine and later designed the ACE (Automatic Computing Engine), an early stored-program computer. Established that a machine can execute any computable process via instructions in memory.
Source: Encyclopaedia Britannica, “Alan Turing.”

3. Logical and informational foundations (1930s–1940s)

Claude Shannon: Showed that electrical circuits can implement Boolean logic and founded information theory. This made digital circuit design precise and scalable.
Source: Encyclopaedia Britannica, “Claude Shannon.”

4. Stored-program architecture (mid-1940s)

John von Neumann: Formalized the stored-program concept, where instructions and data share memory. This became the standard architecture for general-purpose computers.
Source: Encyclopaedia Britannica, “stored-program concept.”

5. Early electronic and commercial computing + software abstraction (1940s–1950s)

Grace Hopper: Worked on UNIVAC I and developed early compilers. Helped move programming from machine-specific code to higher-level abstraction.
Source: Encyclopaedia Britannica, “Grace Hopper.”

6. Transistor replaces vacuum tubes (1947 onward)

John Bardeen, Walter Brattain, William Shockley: Invented the transistor. Enabled smaller, more reliable, lower-power computers.
Source: Nobel Prize, Physics 1956 summary.

7. Integrated circuit (late 1950s–1960s)

Jack Kilby: Demonstrated the first integrated circuit.
Robert Noyce: Developed a practical silicon-based IC with interconnects.
Shift from discrete components to monolithic chips.
Source: Computer History Museum.

8. Scaling law and industry direction (1965 onward)

Gordon Moore: Predicted exponential growth in transistor density (“Moore’s law”). Provided a planning target for semiconductor development.
Source: Computer History Museum.

9. Microprocessor (early 1970s)

Ted Hoff, Federico Faggin (Intel 4004): Put a CPU onto a single chip. Turned the computer into a component that can be embedded in other systems.
Source: Computer History Museum.

10. Processor architecture optimization (1980s onward)

John Hennessy, David Patterson: Developed RISC architecture principles. Simplified instruction sets to improve performance and efficiency.
Source: ACM A.M. Turing Award materials.

Computer engineering is the branch of engineering that focuses on computer hardware and software. Integrates electrical engineering and computer science.
Computers are made out of electricity and silicon. The basic component of all computers is the transistor, which is simply an electric switch. A bunch of transistors
connected together create logic gates. A bunch of logic gates together can perform mathematical operations, and a bunch of those together form the core of a CPU, which
is the brain of every modern computer known to man. CPUs have control units that interpret commands called opcodes and direct other components of the CPU, arithmetic logic units that perform arithmetical and logical operations, and cache memory for quickly storing the results of intermediate computations while processing, or the location of the next command to execute. A computer at heart is simply giving a CPU access to more memory in the form of RAM or disk, perhaps GPUs for heavy image or data processing or artificial intelligence, and peripherals for graphics, control, input, and output.

Before integrated circuits, electrical circuits were separate devices/components that all had to be wired together. This means they took up a lot of space, and as a circuit became more complex, the number of wires and connections grew exponentially. This was called the tyranny of numbers. It was buggy, messy, and a hard limit on the complexity and size of machines. Integrated circuits took all of the components and instead carved them into one piece of silicon or semiconductor material, allowing the work of an entire circuit to be completed by one component. Microprocessors take this one step further. The first CPUs would take memory, logic, and arithmetic modules - all separate integrated circuit parts - and wire them together, running into the same issue. The microprocessor put the entire CPU on one chip.

### Insulin

**Etymology:** From the Latin *insula*, meaning "island." The name derives from the "islets of Langerhans" — little clumps of tissue in the pancreas that secrete hormones like insulin. These were originally observed by Paul Langerhans, a German medical student, in 1869.

Insulin is an **anabolic hormone**, meaning it regulates the construction of more complex molecules from simpler ones, rather than the other way around. Anabolism — the process of synthesizing larger molecules from smaller ones — is the opposite of catabolism, breaking down complex molecules into simpler ones. Together, they form the two functions of metabolism.

Encoded in humans by the **INS gene**.

**Function:** Regulates the management of carbohydrates, fats, and protein by the body. Insulin encourages absorption of glucose from the blood into cells of the liver, fat, and skeletal muscles.

> *Q: How does this occur?*

**History of Insulin:**

- George Ludwig Zuelzer achieved partial success in treating dogs with pancreatic extract.
- Nicolae Paulescu developed an aqueous pancreatic extract that normalized blood sugar of a diabetic dog. Published *Research on the Role of the Pancreas in Food Assimilation*.

### Merkle Trees

Tree data structures have two types of nodes: **leaves** and **inner nodes**. Leaves are nodes with no children — nodes at the very bottom of the tree. Inner nodes are nodes that have at least one child node.

In a **Merkle tree**, the leaves are hashes of some type of data, while inner nodes contain hashes of their child nodes. In essence, the leaves contain hashes of some data, their parents contain hashes of the leaves, *their* parents contain hashes of those hashes, and so on — until the root of the tree contains a hash of the entire tree. This creates a chain of hashing and integrity throughout the tree. It would be impossible to change the data in the leaves without invalidating the entire tree up to the root.

This provides computational advantages, since trees are more time- and space-efficient than list-oriented structures. Merkle trees allow efficient and secure verification of the contents of a large data structure. Due to their logarithmic complexity, verifying that a particular piece of data is part of a Merkle tree requires computing an amount of hashes proportional to the logarithm of the number of leaves in the tree.

Another way of saying all of the above: the root node of a Merkle tree is a *commitment*, and Merkle trees allow efficient verification that certain leaf nodes are part of that commitment.

**Merkle trees** are named after Ralph Merkle, their inventor, who patented the concept in 1979.
